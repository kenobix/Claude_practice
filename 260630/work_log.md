# 作業ログ: システムプロンプトでチューニングしたAIチャットの作成（2026-06-30）

## 概要

[260521/gemini.html](../260521/gemini.html)をベースに、固定のシステムプロンプト（プリプロンプト）を裏側で自動的にAPIへ同梱し、ユーザーには意識させずに「特定分野の専門家AI」として振る舞わせるチャットアプリを作成した。

成果物: [index.html](./index.html)（生成AIシステムアーキテクト相談室）

---

## やりたかったこと

汎用的なAIを「チューニング」する仕組み、すなわち会話開始前にあらかじめプリプロンプト（システムプロンプト）をAPIに渡しておき、ユーザー側はその存在を意識せずに使えるチャットシステムを作りたい、という相談からスタートした。

---

## 設計方針の検討

### システムプロンプトの注入方法

Gemini APIの`generateContent`リクエストには`systemInstruction`パラメータがあり、ここに固定のテキストを渡すことで、会話全体を通してAIの役割・口調・回答方針を指定できる（[260521/README.md](../260521/README.md)で先にまとめていた内容を活用）。

```json
{
  "systemInstruction": { "parts": [{ "text": "（固定のプリプロンプト）" }] },
  "contents": history
}
```

`history`（会話履歴）とは別フィールドなので、ユーザーの入力欄や設定画面には一切出さずに、JS内に定数として埋め込み、送信時に毎回付与するだけで実現できる。

### トレードオフの確認

フロントエンドのJSにシステムプロンプトを直書きする方式は、ブラウザの「ソースを表示」で誰でも内容を読めてしまう（APIキーのように秘匿はできない）。今回は学習目的でネット公開しても問題ない内容のため、この方式を採用した。秘匿が必要な場合はサーバー/プロキシ経由の構成が必要になる点を確認済み。

専門家AIの分野をユーザーに確認した結果、「生成AIに詳しいシステムアーキテクトに相談できる専門家」に決定。

- LLM・RAG・エージェント・ベクトルDB・プロンプトエンジニアリングに精通したシステムアーキテクトとして振る舞う
- 選択肢が複数ある場合はトレードオフ（コスト・精度・レイテンシ・運用負荷など）を整理して提示する
- 具体的な技術名・サービス名（ChromaDB, FAISS, Pinecone, LangChain, Gemini APIなど）を必要に応じて挙げる
- 抽象的な一般論で終わらせず、実装や構成のイメージが湧く提案をする
- 日本語で簡潔に回答する

## 実装

[260521/gemini.html](../260521/gemini.html)の構成（設定パネル・メッセージ表示・タイピングインジケーターなど）をそのまま流用し、以下のみ変更:

| 変更点 | 内容 |
|--------|------|
| `SYSTEM_PROMPT`定数を追加 | JS内に直書き。設定画面には表示しない（編集・閲覧不可） |
| `fetch`のリクエストボディ | `systemInstruction`フィールドを追加し、`SYSTEM_PROMPT`を毎回同梱 |
| UI文言 | タイトル・welcome文を「生成AIシステムアーキテクト相談室」向けに変更 |

APIキーの保存方式（`localStorage`のみ、サーバー送信なし）は元のgemini.htmlと同じ。

## 発生した問題と対応

| 問題 | 原因 | 対応 |
|------|------|------|
| `git push`が認証エラー（`Invalid username or token`） | GitHub PATの有効期限切れ・未設定 | ユーザーがトークンを更新 |
| `git push`が`fatal: could not read Username`で失敗 | `~/.git-credentials`が空でcredential helperが資格情報を読めない非対話環境 | `git push https://<username>:<token>@github.com/...`形式で直接認証情報を渡す方法を案内 |

## 教訓

| # | 教訓 |
|---|------|
| 1 | Gemini APIの`systemInstruction`は`contents`（会話履歴）とは独立したフィールドなので、UIに出さずにJS内で固定値として注入するだけで「ユーザーに意識させないプリプロンプト」が実現できる |
| 2 | フロントエンドのみの構成では、システムプロンプトはAPIキーと違い「秘匿」はできない（ソースを見れば読める）。公開して問題ない内容かどうかを事前に確認しておく必要がある |
| 3 | credential.helperが`store`設定でも、非対話シェルではユーザー名の入力プロンプトを出せず失敗する。トークンをURLに埋め込んで`git push https://user:token@host/repo.git`の形で直接pushすると、credential helperにも保存されて以降は`git push`だけで通る |
