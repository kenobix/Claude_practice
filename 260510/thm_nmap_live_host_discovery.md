## Task 2: Subnetworks（サブネットワーク）

### ネットワークの基本用語

| 用語 | 定義 |
|------|------|
| **Network Segment（ネットワークセグメント）** | EthernetスイッチやWi-Fiアクセスポイントなど、共有媒体で接続されたコンピュータのグループ。**物理的**な接続単位 |
| **Subnetwork / Subnet（サブネット）** | 同一ルーターを使うよう設定された1つ以上のセグメントをまとめた論理的なグループ。**論理的**な接続単位 |

ネットワークセグメントは物理、サブネットは論理という点が核心的な違い。

### サブネットのCIDR表記とホスト数

| CIDR | サブネットマスク | 最大ホスト数 | 用途例 |
|------|----------------|-------------|--------|
| `/16` | 255.255.0.0 | 約65,000台 | 大規模ネットワーク |
| `/24` | 255.255.255.0 | 約250台 | 一般的なオフィス・家庭LAN |

### ARPとサブネットの関係

**ARP（Address Resolution Protocol）とは**
- IPアドレスからMACアドレス（ハードウェアアドレス）を解決するプロトコル
- データリンク層（Layer 2）のプロトコル
- ARPクエリに応答があれば「そのホストはオンライン」と判断できる

**ARPの重要な制約**

```
ネットワークA (10.1.100.0/24)        ネットワークB (10.1.200.0/24)
[computer1] [computer2] [computer3]  [computer4] [computer5] [computer6]
        |           |                        |           |
      [switch1]----[router]---------------[switch2]
```

- **同一サブネット内**: ARPクエリは届く → ライブホストを検出できる
- **別サブネット**: ARPパケットはルーターを越えない（ルーティング不可）
- → **異なるサブネットのホストにはARPスキャンが使えない**

ARPはリンク層プロトコルであり、パケットはそのサブネット内にバインドされる。別サブネットへのパケットはデフォルトゲートウェイ（ルーター）経由で転送されるが、ARPクエリだけは越えられない。

#### 演習1：computer1 → broadcast（computer1あて）でARP Request、Data: computer6

| 質問 | 答え | 解説 |
|------|------|------|

| computer6はARPリクエストを受信したか？ | **N（受信せず）** | computer6は別サブネット（switch2配下）のため届かない |

| 質問 | 答え | 解説 |
|------|------|------|
| ARPリクエストを受信できたデバイス数は？ | **4台** | 同一サブネット（switch2配下）のcomputer4〜6＋switch2 |
| computer6はARPリクエストに応答したか？ | **Yea（応答した）** | computer6はcomputer4と同一サブネット内のため到達・応答可能 |

→ 送信元が**同じサブネットに属しているか**が決定的な差になる。

## Task 3: Understanding Hosts Discovery through TCP/IP Layer

### ホスト発見に使えるプロトコル（レイヤー別）

| レイヤー | プロトコル | 用途 |
|----------|----------|------|
| Link Layer（リンク層） | **ARP** | MACアドレス解決 → 同一サブネット内のホスト発見 |
| Network Layer（ネットワーク層） | **ICMP** | Ping（Echo）によるホスト到達確認 |
| Transport Layer（トランスポート層） | **TCP** | 特定ポートへのパケット送信でホスト確認 |
| Transport Layer（トランスポート層） | **UDP** | 同上。ICMPがブロックされている場合に有効 |

### 各プロトコルの動作

**ARP**
- ブロードキャストフレームをネットワークセグメントに送り、特定IPのMACアドレスを問い合わせる
- 応答があれば「オンライン」と判断できる
- 同一サブネット内のみ有効

**ICMP**
- Type 8（Echo Request）と Type 0（Echo Reply）を使用するのがpingの基本
- **同一サブネット内にpingを送る場合、ICMPの前にARPクエリが実行される**

**TCP / UDP**
- ICMPがブロックされている環境でも、一般的なポート宛に細工したパケットを送ることでホストの存在確認が可能

---

### ネットワークシミュレーター演習（クイズ）

#### 演習1：computer1 → computer3 へ Ping Request

| 質問 | 答え | 解説 |
|------|------|------|
| pingの前にcomputer1が送ったパケットの種類は？ | **ARP Request** | ICMPの前にMACアドレス解決が必要なため自動でARP送信 |
| pingを送る前にcomputer1が受け取ったパケットの種類は？ | **ARP Response** | computer3がMACアドレスを返答 |
| pingリクエストに応答したコンピュータ数は？ | **1台** | computer3のみ応答 |

#### 演習2：computer2 → computer5 へ Ping Request（異なるサブネット間）

| 質問 | 答え | 解説 |
|------|------|------|
| 最初のARP Requestに最初に応答したデバイスは？ | **router** | 別サブネット宛なのでデフォルトゲートウェイ（ルーター）のMACを解決する |
| 2番目のARP Requestに最初に応答したデバイスは？ | **computer5** | ルーターが宛先サブネット側でcomputer5のMACを解決する |
| 2回目のPing Requestでも新しいARPリクエストは必要だったか？ | **nay（不要）** | ARPはキャッシュされるため、2回目以降はARPなしで送信できる |

**ポイント：** 異なるサブネットへのpingでは、ARP解決が2段階（ルーターのMAC → 宛先のMAC）になる。ただしARPキャッシュにより2回目以降は省略される。

## Task 4: Enumerating Targets（スキャン対象の指定方法）

### nmap のターゲット指定構文

| 指定方法 | 例 | 説明 |
|---------|-----|------|
| **リスト（スペース区切り）** | `192.168.1.1 scanme.nmap.org example.com` | 複数のIPやホスト名を並べる |
| **ハイフン範囲** | `10.11.12.15-20` | 15〜20の6アドレスをスキャン |
| **CIDR表記** | `192.168.1.0/30` | サブネット全体をスキャン（4アドレス） |
| **ファイル指定** | `nmap -iL hosts.txt` | ファイルに書いたリストを読み込む |
| **オクテット範囲** | `10.10.0-255.101-125` | 各オクテットに範囲を指定できる |

### よく使うオプション

| オプション | 動作 |
|-----------|------|
| `nmap -sL TARGETS` | **実際にスキャンせず**スキャン対象のホスト一覧だけ表示する（逆引きDNSも試みる） |
| `nmap -sL -n TARGETS` | `-n` を追加するとDNS解決をスキップして高速化 |

`-sL` はスキャン前に「何台を対象にするか」を確認するのに便利。誤爆防止に使う。

---

### クイズ

| 質問 | 答え | 解説 |
|------|------|------|
| `10.10.12.13/29` の場合、nmapが最初にスキャンするIPは？ | **10.10.12.8** | /29はネットワークアドレスが8の倍数。13が属するブロックは10.10.12.8〜15 |
| `10.10.0-255.101-125` の場合、スキャン対象のIP数は？ | **6400** | 256（0〜255）× 25（101〜125）= 6,400 |

#### /29 の計算メモ

```
/29 = サブネットマスク 255.255.255.248
ブロックサイズ = 256 - 248 = 8
10.10.12.13 が含まれるブロック: 10.10.12.8 〜 10.10.12.15
  → ネットワークアドレス: 10.10.12.8  ← これがnmapの開始アドレス
  → ブロードキャスト:     10.10.12.15
  → ホスト使用可能:       10.10.12.9 〜 10.10.12.14（6台）
```

## Task 5: Nmap Host Discovery Using ARP

### Nmapのデフォルト動作（オプション未指定時）

| 状況 | Nmapの動作 |
|------|-----------|
| **特権ユーザー** ＋ **ローカルネットワーク** | ARP リクエストを使用 |
| **特権ユーザー** ＋ **外部ネットワーク** | ICMP Echo + TCP ACK (port 80) + TCP SYN (port 443) + ICMP Timestamp |
| **非特権ユーザー** ＋ **外部ネットワーク** | TCP 3-way handshake（SYN to port 80 & 443） |

特権ユーザー = root または sudo 実行可能なユーザー。

### ARP スキャンのコマンド

```bash
# ARPスキャンのみ（ポートスキャンなし）
nmap -PR -sn 10.200.6.0/24

# -PR : ARP pingのみ使用
# -sn : ポートスキャンをしない（ホスト発見のみ）
```

### ARP スキャンの仕組み

1. Nmapがサブネット内の全IPにブロードキャストでARPリクエストを送信
2. オンラインのホストがARP Replyで自分のMACアドレスを返す
3. 返答があればそのホストは「生存」と判断

Wiresharkで見ると：送信元MAC = AttackBox、宛先MAC = ブロードキャスト（FF:FF:FF:FF:FF:FF）でIPアドレスを順番に問い合わせている様子が確認できる。

### ARPスキャンの用途

- **ポスト・エクスプロイテーション（侵入後）の内部列挙**に特に有効
- ARPはLayer 2で動作しファイアウォールにフィルタリングされにくい
- ローカルセグメント内のホストを迅速・確実に特定できる

### クイズ

| 質問 | 答え |
|------|------|
| `CONNECTION_IP/24` をスキャンして発見されたライブホスト数は？ | **1台** |

## Task 6: Nmap Host Discovery Using ICMP

### ICMPを使ったホスト発見の3種類

| オプション | ICMPタイプ | 動作 | 検出結果（例） |
|-----------|-----------|------|--------------|
| `-PE` | Type 8 (Echo Request) / Type 0 (Echo Reply) | 最も基本的なping | 3台検出 |
| `-PP` | Type 13 (Timestamp Request) / Type 14 (Timestamp Reply) | Echo がブロックされた場合の代替 | 2台検出 |
| `-PM` | Type 17 (Address Mask Request) / Type 18 (Address Mask Reply) | 別の代替手段。多くの環境でブロックされる | 0台（ブロックされた） |

### コマンド例

```bash
# ICMP Echo
sudo nmap -PE -sn 10.200.6.0/24

# ICMP Timestamp
nmap -PP -sn 10.200.6.0/24

# ICMP Address Mask
nmap -PM -sn 10.200.6.0/24
```

### 重要な注意点

- **Windowsの新バージョン**はデフォルトでICMP Echoをブロックするホストファイアウォールを持つ
- 同一サブネット内が対象の場合、ICMPの前にARPクエリが自動的に実行される
- **1つの手法がブロックされても代替手法で試みることが重要**（-PE → -PP → -PM と切り替える）
- `-PM` が0台を返しても、他スキャンで2台確認できていれば「ファイアウォールでブロックされている」と判断できる

### クイズ

| 質問 | 答え |
|------|------|
| ICMP Timestamp を使うオプションは？ | **`-PP`** |
| ICMP Address Mask を使うオプションは？ | **`-PM`** |
| ICMP Echo を使うオプションは？ | **`-PE`** |

## Task 7: Nmap Host Discovery Using TCP and UDP

### TCP SYN Ping（`-PS`）

SYNフラグを立てたパケットを送り、応答でホスト存在を確認する。

| ポートの状態 | 応答 | 判断 |
|-------------|------|------|
| オープン | SYN/ACK | ホスト生存 |
| クローズ | RST | ホスト生存 |
| 無応答 | なし | オフライン or フィルタリング |

```bash
# デフォルト（port 80）
nmap -PS -sn 10.200.6.0/24

# ポート指定
nmap -PS21        # port 21のみ
nmap -PS21-25     # port 21〜25
nmap -PS80,443,8080  # 3ポート指定
nmap -PS23        # Telnet port（クイズの答え）
```

**特権 vs 非特権の違い：**
- **特権ユーザー（root/sudo）**: SYNを送り、SYN/ACKを受け取っても3-way handshakeを完了させない（ステルス性が高い）
- **非特権ユーザー**: ポートがオープンなら3-way handshakeを完了せざるを得ない

### TCP ACK Ping（`-PA`）

ACKフラグを立てたパケットを送る。**特権ユーザー必須**。

```bash
sudo nmap -PA -sn 10.200.6.0/24

# ポート指定（デフォルト port 80）
# -PA21, -PA21-25, -PA80,443,8080
```

- 既存のコネクションに属さないACKパケットを受け取ったホストは**必ずRSTで応答する**
- このRST応答でホストの生存を確認する
- 非特権ユーザーが実行すると自動的に3-way handshakeを試みる

### UDP Ping（`-PU`）

```bash
sudo nmap -PU -sn 10.200.6.0/24
```

- UDPパケットを**クローズしているUDPポート**に送る
- クローズポートからは `ICMP Port Unreachable` エラーが返る → ホスト生存と判断
- オープンポートは通常何も返さないため、**あえてクローズポートを狙う**のがポイント

### Masscan（高速スキャナー）

Nmapと似たアプローチだが、パケット生成レートが非常に高く**高速スキャン**が可能。

```bash
masscan 10.200.6.0/24 -p443
masscan 10.200.6.0/24 -p80,443
masscan 10.200.6.0/24 -p22-25

# インストール（未インストールの場合）
apt install masscan
```

### ホスト発見オプション まとめ

| オプション | 手法 | 特権必要 | デフォルトポート |
|-----------|------|---------|----------------|
| `-PR` | ARP ping | 不要（ローカルのみ） | — |
| `-PE` | ICMP Echo | 不要 | — |
| `-PP` | ICMP Timestamp | 不要 | — |
| `-PM` | ICMP Address Mask | 不要 | — |
| `-PS` | TCP SYN ping | 不要（非特権は3WHS完了） | 80 |
| `-PA` | TCP ACK ping | **必要** | 80 |
| `-PU` | UDP ping | **必要** | — |

### クイズ

| 質問 | 答え | 理由 |
|------|------|------|
| 特権不要のTCP pingスキャンは？ | **TCP SYN Ping** | 非特権でも実行可能（ただし3WHS完了） |
| 特権必要のTCP pingスキャンは？ | **TCP ACK Ping** | ACKパケット送信には特権が必要 |
| Telnetポートに TCP SYN Ping を送るオプションは？ | **`-PS23`** | Telnet = port 23 |
